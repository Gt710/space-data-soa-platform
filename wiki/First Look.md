## Актуальність — аргументи для диплому

Це найважливіша частина вступу. Ось три кути, які переконують комісію:

**1. Глобальний контекст — космічна галузь вибухово зростає**

З 2020 року кількість активних супутників на орбіті зросла втричі (Starlink, OneWeb та ін.). Станом на 2025 рік на орбіті понад 9 000 активних об'єктів, і відстежити їх рухом без спеціалізованих інструментів неможливо. Попит на інструменти моніторингу і аналізу космічних даних зростає пропорційно.

**2. Проблема фрагментованості даних**

NASA, ESA, NOAA, Space-Track — кожен портал існує окремо, має власний формат, власну автентифікацію, часто незручний інтерфейс. Дослідник, якому потрібні орбіта супутника + стан сонячної погоди + знімки місії, мусить переходити між трьома-чотирма платформами і вручну зіставляти дані. Це і є та проблема, яку вирішує платформа.

**3. SOA як архітектурне рішення — чому саме воно**

Сервіс-орієнтована архітектура дозволяє масштабувати окремі сервіси незалежно (наприклад, під час сонячного спалаху різко зростає навантаження саме на Space Weather, а не на всю систему), легко підключати нові джерела даних, і надавати API стороннім розробникам — що перетворює платформу з інструменту на інфраструктуру.

## Існуючі аналоги

### Платформи відстеження супутників

**Heavens-Above** — надає точні прогнози прольотів МКС, видимих супутників і радіосупутників, інтерактивні карти зоряного неба та режим нічного спостереження. Існує з 1999 року, безкоштовний. Мінус — не має AR-трекінгу, push-сповіщень та мобільно-оптимізованого досвіду.

**NASA Earthdata** — охоплює пошук, порівняння, візуалізацію і доступ до всіх наукових даних NASA про Землю. Орієнтована на наукове співтовариство, складна для початківців.

**Copernicus Data Space (ESA)** — пропонує Copernicus Browser для перегляду й аналізу супутникових знімків прямо у браузері, а також openEO — відкритий API для хмарної обробки даних дистанційного зондування. Вузька спеціалізація — Земля, не космос загалом.

## Рекомендований стек — найлегший шлях

### Варіант А: Python-монорепо (найпростіший, рекомендую для диплому)

Оскільки ти знаєш Python — це найшвидший шлях до робочого прототипу:

**Backend:** FastAPI (Python) — один фреймворк для всіх мікросервісів. Кожен сервіс — окрема папка з власним `main.py`. FastAPI автоматично генерує Swagger-документацію (OpenAPI), що ідеально для диплому — одразу є специфікація API.

**Чому FastAPI, а не Flask/Django:**

- Async з коробки — важливо для паралельних запитів до NASA/NOAA
- Автоматичний Swagger UI за адресою `/docs`
- Валідація даних через Pydantic — мінімум коду для перевірки відповідей API
- Найпростіший старт серед Python-фреймворків

**Frontend:** React (JS/TS) — найбільша екосистема готових компонентів для карт і графіків. Якщо не хочеш JS — можна взяти Streamlit (Python!) для швидкого прототипу дашборду.

**БД:** PostgreSQL — одна БД, яку ти вже, імовірно, знаєш. TimescaleDB — це просто розширення через `CREATE EXTENSION timescaledb`, нічого нового вчити не треба.

### Варіант Б: Kotlin для одного сервісу (використати обидві мови)

Kotlin + Ktor або Spring Boot — для сервісу Satellite Tracker. Це виглядатиме сильніше в дипломі (демонструє SOA: сервіси на різних мовах взаємодіють через API). Решта сервісів — Python.
